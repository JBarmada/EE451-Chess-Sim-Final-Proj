# Chess Monte Carlo Simulation — CARC Setup

Reproducible HPC experiment scripts for the EE451 Final Project chess simulation study.

## Software Environment

| Component | Version |
|-----------|---------|
| GCC / g++ | default via `module load gcc` |
| Python    | default via `module load python` |
| OpenMP    | libgomp bundled with the loaded GCC |
| SLURM     | Discovery cluster scheduler |
| Chess lib | Disservin/chess-library v0.9.4 (header-only, `chess.hpp`) |
| C++ std   | C++17 with `-O3 -DNDEBUG` |

Exact versions are recorded in `carc/build_manifest.txt` after `build.sh` runs.

Python analysis script uses **standard library only** — no pip installs required.

## File Overview

```
carc/
├── build.sh                 # Step 1: compile all binaries
├── submit_all.sh            # Step 2: submit all jobs with dependency chaining
├── run_serial.job           # SLURM: serial baseline (100k, 1m, 10m, 100m)
├── run_openmp_scaling.job   # SLURM: thread scaling (1–32t, 10m games)
├── run_openmp_sizes.job     # SLURM: problem-size throughput (32t, all sizes)

├── analyze_results.py       # Metrics computation + report generation
├── README_carc.md           # This file
├── build_manifest.txt       # Generated by build.sh (gcc version, timestamp)
├── logs/                    # SLURM stdout/stderr (one file per job ID)

    ├── metrics_raw.csv
    ├── speedup_table.csv
    ├── scaling_table.csv
    ├── amdahl_table.csv
    ├── statistical_accuracy.csv
    └── report.md
```

Simulation results (summary.txt files) are written to:
```
GeminiCoder/results/
├── {100k,1m,10m,100m}_serial/summary.txt
└── {100k,1m,10m,100m}_openmp/{1,2,4,8,16,32}/summary.txt
```

## Step-by-Step Reproduction

### 1. Connect to CARC

```bash
ssh <your_usc_id>@discovery.usc.edu
```

### 2. Navigate to the project

```bash
cd /path/to/chess-library/GeminiCoder/
```

Replace `/path/to/` with your actual path (e.g., `~/projects/EE451/`).

### 3. Compile all binaries

```bash
module load gcc
bash carc/build.sh
```

Expected output: 8 binaries created (`sim100k_serial`, `sim1m_serial`, ..., `sim100m_openmp`).
A `carc/build_manifest.txt` file is written with compiler provenance.

### 4. Submit all jobs

```bash
cd carc/
bash submit_all.sh
```

This submits 4 jobs and prints their IDs. The analysis job has `afterok` dependency
on all three compute jobs — it runs automatically when they all succeed.

### 5. Monitor progress

```bash
squeue -u $USER --format="%.10i %.20j %.8T %.10M %.12l"
```

### 6. Check results

After the analysis job completes:

```bash
cat carc/results/report.md
```

CSVs for plotting/further analysis are in `carc/results/*.csv`.

## What Each Job Does

| Job | Partition | Resources | Wall Time | Purpose |
|-----|-----------|-----------|-----------|---------|
| `run_serial.job` | main | 1 CPU, 8 GB | 4 hr | Serial baseline at all 4 sizes |
| `run_openmp_scaling.job` | epyc-64 | 32 CPU, 64 GB, exclusive | 1 hr | Thread scaling 1–32 at 10M games |
| `run_openmp_sizes.job` | epyc-64 | 32 CPU, 64 GB, exclusive | 30 min | Throughput at 32t for all 4 sizes |
| `run_analyze.job` | main | 1 CPU, 4 GB | 15 min | Parse results, compute metrics |

The OpenMP jobs use `--exclusive` to prevent co-located workloads from contaminating
thread-count measurements. Without exclusive access, parallel efficiency numbers
would be artificially deflated.

| Thread-count scalability | Computed | `scaling_table.csv` |
| Problem-size scalability | Computed | `speedup_table.csv` |

### Load Imbalance Note

The `openMP.cpp` code uses `schedule(static)`, meaning each thread gets an equal
**count** of games but not necessarily equal **work** (game lengths vary from ~10 to
~5897 plies). For ≥1M games, the Central Limit Theorem makes per-thread total work
nearly uniform, making static scheduling near-optimal. This is discussed in `report.md`.

## Reproducibility Notes

**Statistical (outcomes):** Game results are NOT bit-for-bit reproducible because
the serial/OpenMP binaries seed from `std::random_device`. However, at ≥10M games:
- Checkmate rate converges to within ±0.05% of 15.55%
- Draw/win rates converge to within ±0.1%
- The z-score vs Labelle baseline should be |z| < 2 at 10M, |z| < 1 at 100M

**Performance (timing):** Reproducible to within ~5% on identical partition/node type.
`--exclusive` allocation removes the main source of variance (node sharing).
The `carc/logs/run_*.{jobid}.out` files record timestamps and job IDs for traceability.

**Rerunning:** Resubmitting jobs overwrites previous `summary.txt` files. SLURM
stdout logs preserve the run history via unique job IDs in filenames.

## Troubleshooting

**Binary not found:** Run `bash carc/build.sh` from `GeminiCoder/` before submitting.

**Job fails with OOM:** The 100M OpenMP run requires ~400 MB for `gameLengths` vector.
If your node has less memory, reduce `--mem` is fine down to `16G`; the 64G request
is conservative. If you see actual OOM kills, reduce the problem size.

**`epyc-64` partition unavailable:** Replace with `--partition=main` and add
`--constraint=epyc` if available, or remove `--exclusive` and accept that thread
scaling numbers will have more variance.

**Analysis job cancelled (dependency failed):** If any compute job fails, the
analysis job is cancelled automatically. Fix the compute job and resubmit just
`run_analyze.job` manually: `sbatch carc/run_analyze.job`

**Partition names changed:** Check available partitions with `sinfo` and update
`#SBATCH --partition=` lines accordingly.
